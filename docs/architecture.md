## アーキテクチャパターン

### パイプラインアーキテクチャ（一方向データフロー）

本ツールは変換パイプラインとして設計する。各ステージは前段の出力のみに依存し、後段を知らない。

```
┌──────────────────────────────────────────────────────────────┐
│  ブラウザ（HTML5 + Vanilla JS）[P2 Web UI]                   │
│  - D&Dエリア、ファイル選択                                     │
│  - fetch() API で POST /api/convert                          │
│  - レスポンス（.md / .zip）を自動ダウンロード                   │
└────────────────────┬─────────────────────────────────────────┘
                     │ HTTP multipart/form-data
┌────────────────────▼─────────────────────────────────────────┐
│  Web UIレイヤー（web/app.py）[P2 Web UI]                      │
│  - FastAPI: POST /api/convert、GET /health                   │
│  - アップロードファイルをtempfileに書き出し変換パイプラインを呼出│
│  - 単一ファイル → text/markdown、複数 → application/zip       │
│  - 静的ファイル配信（/static/ → web/static/index.html）       │
└────────────────────┬─────────────────────────────────────────┘
                     │ Path（tempfile）
┌────────────────────▼─────────────────────────────────────────┐
│  CLIレイヤー（cli.py / __main__.py）                          │
│  - 引数パース・バリデーション                                  │
│  - パイプライン起動・シート間統合                              │
│  - run_file(): 1ファイル変換ヘルパー（Web UIからも呼び出し可）  │
└────────────────────┬─────────────────────────────────────────┘
                     │ path, options
┌────────────────────▼─────────────────────────────────────────┐
│  読み込みレイヤー（reader/）                                   │
│  - xlsx_reader: openpyxl → list[RawCell]                     │
│  - xls_reader:  xlrd    → list[RawCell]  (P1)                │
│  印刷領域フィルタ・非表示行列除外・結合セル処理                 │
└────────────────────┬─────────────────────────────────────────┘
                     │ list[RawCell]
┌────────────────────▼─────────────────────────────────────────┐
│  空間解析レイヤー（parser/cell_grid.py, merge_resolver.py）   │
│  - CellGrid: col_unit・modal_row_height を算出               │
│  - MergeResolver: RawCell → TextBlock（空白スキップ）         │
└────────────────────┬─────────────────────────────────────────┘
                     │ list[TextBlock], CellGrid
┌────────────────────▼─────────────────────────────────────────┐
│  構造検出レイヤー（parser/structure_detector.py,             │
│                   parser/table_detector.py）                  │
│  - インデントティア計算                                        │
│  - 見出し/段落/リスト/ラベル:値/BLANK に分類                   │
│  - グリッド表を検出してTableElementに変換                      │
└────────────────────┬─────────────────────────────────────────┘
                     │ list[DocElement]
┌────────────────────▼─────────────────────────────────────────┐
│  出力レイヤー（renderer/markdown_renderer.py）                 │
│  - DocElement → Markdown文字列                               │
│  - 脚注（セルコメント）の追加                                  │
│  - 空行圧縮・後処理                                           │
└──────────────────────────────────────────────────────────────┘
```

**レイヤー間の制約**:
- 上位レイヤーは下位レイヤーに依存しない（Readerはrendererを知らない）
- データモデル（`models.py`）はすべてのレイヤーが参照する共有レイヤー
- 各コンポーネントはpure function（副作用なし）として実装し、テスト容易性を確保

> パイプライン図中の `col_unit`・`modal_row_height`・インデントティアの定義は [用語集](./glossary.md) を参照。

### エラーハンドリング戦略

- **例外はCLIレイヤーで集約する**: パーサー・レンダラー層は例外を上位に伝播させ、CLIレイヤーが一括してキャッチして適切な終了コードに変換する
- **ユーザー向けメッセージ**: CLIレイヤーで `stderr` に日本語エラーメッセージを出力し、exitコード1で終了
- **想定外エラー**: `except Exception` で捕捉してスタックトレースを表示し、exitコード2で終了

## データ永続化戦略

### ストレージ方式

| データ種別 | ストレージ | フォーマット | 理由 |
|-----------|----------|-------------|------|
| 入力Excel | ファイルシステム（読み取り専用） | .xlsx / .xls | ユーザー提供の変換元 |
| 出力Markdown | ファイルシステム | UTF-8 テキスト (.md) | テキストベースで検索・Git管理・AIへの入力が容易 |
| 中間データ | メモリ上のみ | Python dataclass | 変換はステートレス。ファイルへの中間書き出しは不要 |

### 出力ファイルの上書き挙動

本ツールはバックアップ機能を持たない。既存ファイルの保護はユーザーのバージョン管理（Git等）に委ねる。

- 出力先ファイルが既存の場合は上書きする（`-o` オプション指定時のみ発生）
- 入力Excelは読み取り専用で操作し、一切変更しない

## パフォーマンス要件

### レスポンスタイム

| 操作 | 目標時間 | 測定環境 |
|------|---------|---------|
| A4縦1ページ（100行×50列） | 3秒以内 | CPU Core i5相当、メモリ8GB |
| 大規模ファイル（1,000行×50列） | 30秒以内 | 同上 |

### リソース使用量

| リソース | 上限 | 理由 |
|---------|------|------|
| メモリ | 500MB | 1,000行×50列のExcelと中間データをメモリ上で保持する際の上限見積もり |
| ディスク | 入力ファイルと同程度 | 中間ファイルは生成しない |

### パフォーマンス最適化方針

- **openpyxlのread-onlyモードは使用しない**: `ws.merged_cells.ranges` は通常モードのみ対応
- **一括読み込み**: セルをイテレーションは1回（行→列の順）でRawCellを全生成し、後段はメモリ上で処理
- **不要なセルは早期除外**: 非表示・印刷領域外・値なしは読み込み段階で除外し、後段のデータ量を削減

## セキュリティアーキテクチャ

### データ保護

- **入力検証**: ファイルパスのバリデーション（存在確認・拡張子確認）をCLIレイヤーで実施
- **パストラバーサル**: ユーザー指定パスを `Path.resolve()` で正規化してから利用
- **出力先の安全性**: 出力ファイルを書き込む前にディレクトリの存在を確認
- **Web UIのアップロード検証**: 拡張子バリデーション（.xlsx/.xls のみ）・ファイルサイズ上限50MB。アップロードファイルはtempfileに保存して変換後即削除（サーバーに残さない）。bindアドレスは `127.0.0.1` 固定でローカル専用

### 入力検証

```python
def validate_input(path: Path) -> None:
    if not path.exists():
        raise FileNotFoundError(f"ファイルが見つかりません: {path}")
    if path.suffix.lower() not in {".xlsx", ".xls"}:
        raise ValueError(f"対応していないファイル形式です: {path.suffix}")
```

### セキュリティ制約

- 外部ネットワーク通信なし（完全オフライン動作）
- 機密情報は扱わない（CLIツールとして入出力はユーザーのローカルファイルのみ）
- Excelマクロ・外部リンクは実行・解決しない（`data_only=True`）。信頼されていないファイルからの任意コード実行リスクを排除するためのセキュリティ設計上の決定であり、openpyxlの技術的制約と一致している

## スケーラビリティ設計

### データ増加への対応

**想定最大規模**: 1,000行×100列の方眼紙（企業の詳細設計書レベル）

**対策**:
- `openpyxl` は通常モードで全セルをメモリに展開するため、超大規模（10,000行超）では読み込み時間とメモリが増大する。この規模は現バージョンのスコープ外（[パフォーマンス制約](#パフォーマンス制約) 参照）
- TextBlock・DocElementはPython dataclassで軽量。メモリボトルネックは主にopenpyxlのワークブックオブジェクト

### 機能拡張性

**新フォーマット対応**: reader/ に新リーダーを追加するだけでパイプライン下流は変更不要（Readerのインターフェース: `list[RawCell]` を返す）

**新出力形式対応**: renderer/ に新レンダラーを追加するだけで対応可能（将来のHTML・reStructuredText対応を想定）

**見出し判定カスタマイズ**: `--base-font-size` CLIオプションで閾値を調整可能。将来的にはYAMLコンフィグファイルで細かい閾値を設定する余地がある

## テスト戦略

### ユニットテスト

- **フレームワーク**: pytest
- **対象**: 各コンポーネントを独立してテスト（openpyxlをimportしないで済むコンポーネントは依存なしで単体テスト可能）
- **フィクスチャ生成**: openpyxlでプログラム的にxlsxを生成（バイナリのExcelファイルをリポジトリにコミットしない）
- **カバレッジ目標**: 80%以上

### 統合テスト

- **方法**: xlsxフィクスチャ → パイプライン全体 → Markdownゴールデンファイル比較
- **対象**: 6種の代表パターン（PRD成功指標参照）

### パフォーマンステスト

- **現在**: `time.perf_counter()` でA4/1,000行フィクスチャを計測し、pytest内でアサーション（3秒/30秒の閾値）
- **将来**: `pytest-benchmark` を導入してCI上での閾値超過アラートを自動化する（現時点ではスコープ外）

### CI/CD連携

- **現在**: ローカルで `pytest` を実行してテストを確認する
- **将来**: GitHub Actions等のCI環境でテスト・lint・型チェック・パフォーマンステストを自動実行する構成を検討中

## 技術的制約

### 環境要件

- **OS**: Linux / macOS / Windows（クロスプラットフォーム）
- **Python**: 3.12以上
- **最小メモリ**: 256MB（通常のExcelファイル変換時）
- **ディスク**: 入力ファイルと同程度（中間ファイルなし）
- **外部依存**: openpyxl（必須）、xlrd（P1オプション）

### パフォーマンス制約

- openpyxlは通常モードのみ使用（read-onlyモードでは結合セルAPIが非対応のため）
- 10,000行超のファイルは変換時間・メモリとも保証外（詳細は [スケーラビリティ設計](#スケーラビリティ設計) を参照）

### セキュリティ制約

- パスワード保護されたExcelファイルは変換不可（対応外）
- ネットワーク接続不要・外部APIなし

## 依存関係管理

| ライブラリ | 用途 | バージョン管理方針 |
|-----------|------|-------------------|
| openpyxl | xlsx読み込み | `>=3.1.0,<4.0.0`（マイナー固定、パッチ自動） |
| xlrd | xls読み込み（P1） | `>=2.0.0,<3.0.0`（マイナー固定、パッチ自動） |
| FastAPI | Web APIフレームワーク（P2 Web UI） | `>=0.110.0,<1.0.0`（マイナー固定） |
| uvicorn | ASGIサーバー（P2 Web UI） | `>=0.29.0,<1.0.0`（マイナー固定） |
| python-multipart | ファイルアップロード（P2 Web UI） | `>=0.0.9` |
| pytest | テスト | `>=8.0.0`（最新安定版を追従） |
| pytest-cov | カバレッジ | `>=5.0.0` |
| pytest-benchmark | パフォーマンス計測（将来追加候補） | `>=4.0.0` |
| ruff | lint/format | `>=0.4.0` |
| mypy | 型チェック | `>=1.10.0` |

**方針**:
- 本番依存（openpyxl/xlrd）はメジャーバージョンを固定してAPIの破壊的変更を防ぐ
- Web UI依存（FastAPI/uvicorn）は `[web]` optional dependencyとして分離し、CLIのみの用途ではインストール不要
- 開発依存（pytest/ruff/mypy）は最新の安定版を使用してセキュリティパッチを適用
- `pyproject.toml` で依存を管理し、`pip install -e ".[dev]"` で開発環境をセットアップ。Web UI込みは `pip install -e ".[web,dev]"`
